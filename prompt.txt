# AI Coding Prompt: Close Coverage Gaps to 100% on Critical Paths

Standalone prompt. Use once `pytest-cov` is installed and you have a real `htmlcov` report to work from — this prompt is about closing gaps against real, measured coverage, not adding tests blindly.

---

## ROLE

You are a senior test engineer auditing an existing fraud detection test suite (17 tests currently passing) against its original specification. Your job is not to add tests for the sake of a coverage number — it's to find the specific, real gaps between what's tested and what a production fraud system actually needs verified, then close exactly those gaps.

## OBJECTIVE

1. Measure real coverage (not just "tests pass").
2. Diff what's actually tested against the original acceptance criteria for this module.
3. Close every genuine gap, prioritizing correctness-critical and fallback/degradation paths to 100%, not the whole codebase uniformly.
4. Prove it with real coverage output, not a claim.

---

## STEP 1: MEASURE REAL COVERAGE FIRST

Before writing a single new test, run:

```bash
pip install pytest-cov --break-system-packages  # or into the project's venv, matching existing setup
PYTHONPATH=. pytest tests/ \
  --cov=mpesa_safaricom/fraud_anomaly_detection \
  --cov-report=term-missing \
  --cov-report=html \
  -v
```

Report the actual output: overall coverage percentage, and the specific uncovered line numbers per file from `term-missing`. Do not proceed to Step 2 with assumptions about what's covered — work from this real output.

Add coverage enforcement so this can't silently regress later:

```toml
# pyproject.toml
[tool.pytest.ini_options]
addopts = "--cov=mpesa_safaricom/fraud_anomaly_detection --cov-report=term-missing --cov-fail-under=85"
```

### VelocityDetector
- [ ] Exactly-at-threshold does NOT trigger; one-over DOES trigger (boundary test, not just "triggers when clearly over")
- [ ] Events outside the sliding window are correctly excluded/expired
- [ ] Duplicate `txn_id` replayed does not double-count
- [ ] Redis failure → returns "unknown" state, does not raise
- [ ] **Concurrent writes to the same MSISDN from multiple threads produce a correct count** — this is the one most likely to not exist yet; if it doesn't, write it using `threading` or `pytest-xdist`-style concurrent execution against a real (or realistically-faked) Redis client, not a simple mock that can't expose race conditions

### SimSwapCorrelator
- [ ] Swap within lookback + amount over threshold → triggers
- [ ] Swap within lookback + amount under threshold → does NOT trigger

- [ ] **"No swap on record" and "swap feed stale" produce distinguishable results** — check the actual assertion compares specific result fields (e.g., `result.state != "stale"` vs `result.triggered == False`), not just that both don't trigger

### NightTransactionFlagger
- [ ] Local-time-02:00 flags; same UTC instant that is NOT locally 02:00 does not flag — confirm the test actually varies timezone, not just checks a fixed offset

### MLFraudScorer
- [ ] Model unavailable → neutral result, no exception (covered by `test_ml_scorer_missing_model` — confirm it asserts the *result*, not just that no exception was raised)

### FraudScoreAggregator

- [ ] **Explanation content actually names which checks triggered and why** — confirm `test_aggregator_combines` asserts on the explanation string/list content, not just the final decision value
- [ ] **Interaction effect test**: combined score from two correlated checks (velocity + SIM swap) exceeds either individually — confirm `test_aggregator_interaction` asserts the numeric relationship (`combined_score > max(individual_scores)`), not just that both checks appear in the result

### FraudConfig

### Engine / integration

### MuleAccountCorrelator

### Circuit breaker

### Champion/challenger

Do not rewrite or touch tests already marked CONFIRMED COVERED — this step is surgical, not a rewrite.

## CONSTRAINTS

- Do not pad the suite with trivial tests just to move a percentage number — a test that doesn't assert anything meaningful about behavior is worse than no test, since it creates false confidence.
- Do not touch or "improve" a test already confirmed to cover its target correctly — scope is strictly the gaps identified in Step 2.
- If a line is genuinely unreachable or defensive-only, say so explicitly rather than contorting a test to hit it artificially.
